I Built a Memory System That Grades Its Own Homework

March 2026

The problem I kept running into

I use Claude Code every day. Over time, I accumulated a lot of instructions for it — coding standards, project conventions, things I'd corrected more than once. CLAUDE.md files, rules, skills. The usual.

The problem wasn't capturing instructions. Claude Code is good at that. The problem was that I had no idea which ones were actually working.

Some instructions I'd written months ago were probably stale. Some were so generic they matched everything and helped nothing. Some were genuinely good but had keywords so narrow they almost never surfaced. And a few were actively counterproductive — surfacing constantly but getting ignored every time, wasting context budget on advice the model had learned to tune out.

I was maintaining a growing pile of instructions with no feedback signal. It felt like writing tests that never run.

What I wanted

I wanted a memory system that could answer basic questions about itself:

  • Which memories are actually being followed when they surface?
  • Which ones get ignored every time?
  • Which ones are great but never match anything?
  • Which ones should just be deleted?

And then I wanted it to do something about the answers. Not just report — diagnose and propose fixes. Rewrite a confusing instruction. Broaden the keywords on a hidden gem. Remove noise. Escalate enforcement on something that keeps getting violated.

A closed feedback loop. Learn, measure, fix, repeat.

What I built

Engram is a Claude Code plugin that hooks into every phase of a session:

  1. Extract — learns from corrections and session transcripts, deduplicates against what it already knows
  2. Surface — retrieves relevant memories at each prompt via keyword matching, with a token budget so it doesn't overwhelm the model
  3. Evaluate — tracks whether surfaced memories were followed, contradicted, or ignored
  4. Maintain — diagnoses each memory by effectiveness and proposes targeted fixes

Every memory gets classified into one of four quadrants based on how often it surfaces and how often it's followed:

High effectivenessLow effectiveness
Often surfacedWorking — keepLeech — rewrite or escalate
Rarely surfacedHidden gem — broaden keywordsNoise — remove

The system also proposes its own improvements. When it notices that a keyword like "toml" causes irrelevant matches 93% of the time, it proposes de-prioritizing that keyword for future extractions. When it finds near-duplicate memories, it proposes consolidating them. All proposals go through user review before taking effect.

What it's like to use

In practice, I start a session and engram reminds me I can say /recall to load context from my last session. When I type a prompt, relevant memories surface inline. When the agent finishes, I get a quick triage summary if there are maintenance signals.

Periodically I run /memory-triage and spend a few minutes reviewing what engram has found: "these 6 memories are noise — remove all?", "this memory has 100% effectiveness but only surfaced twice — broaden keywords?". It takes a few minutes and the corpus gets noticeably better each time.

The part that surprised me most was the adaptation pipeline. Engram watches its own feedback patterns and proposes system-level tuning — not just "fix this memory" but "stop using this keyword for extraction because it causes too many false positives." The system is learning how to learn better.

How it compares to built-in memory

Claude Code now has built-in memory and dreaming features. Those are great for general preferences and project context — zero setup, just works.

Engram is for a different use case: instructions where you want accountability. Did the agent actually follow this? If not, why? Is this instruction helping or just burning context tokens? Built-in memory stores what you tell it. Engram measures whether what you stored is working, and fixes it when it isn't.

They're complementary. I use both.

Where it's at

Engram is alpha. I've been using it daily for a few weeks and it's been genuinely useful, but APIs and data formats may still change. It's written in pure Go with no CGO dependencies, uses TF-IDF and BM25 for text matching, and calls the Anthropic API only for LLM-dependent operations like classification and extraction.

The whole thing is open source: github.com/toejough/engram

If you use Claude Code and have opinions about how memory should work for LLM agents, I'd love to hear from you. Issues and feedback are welcome.
