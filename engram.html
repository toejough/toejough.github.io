Getting Claude to Learn and Remember

March 2026

This post is from March 2026 and describes an earlier version of engram. A lot has changed since — see the April update for what I tore out and what replaced it.

The problem I kept running into

I use Claude Code every day. Early on, I'd say things like "remember that" or "never do X" and Claude would respond with "good call, I'll remember that for next time" — and then promptly forget. So I started saying "write that down somewhere," then "write that down in CLAUDE.md." Over time, CLAUDE.md filled up with a grab bag of instructions — coding standards, project conventions, things I'd corrected more than once. Some stale, some project-specific, all piling up.

Every new session had a different idea of what to capture and where to put it. Sometimes Claude would deduplicate, sometimes not. Stale instructions layered in next to contradictory ones, and CLAUDE.md was getting large enough for context poisoning — the instructions were starting to make things less effective rather than more. I started extracting parts out into skills. Sometimes that was valuable, sometimes it was overkill. And writing good skills is a ... skill ... in itself.

I wanted a memory system that wasn't overwhelming the context window, that automatically picked up on corrections and important patterns and things worth remembering.

So that's what I built. Along the way I kept fixing problems through trial and error — adding features I thought were useful, pulling them back as the system got too complex. I'm still in that churn, but it's working well enough at this point that it felt like a decent time to share it.

What I wanted

Beyond the basics, I wanted it to be self-correcting — but keep the user in control. It should track what's working and what isn't, diagnose problems, and propose fixes. But never apply changes without asking first.

Rewrite a confusing instruction. Broaden the keywords on a hidden gem. Remove noise. Escalate enforcement on something that keeps getting violated. All proposed, all reviewed by me before taking effect.

A closed feedback loop. Learn, measure, propose, fix. Repeat.

What I have so far

Engram is a Claude Code plugin that hooks into every phase of a session:

  1. Extract — learns from corrections and session transcripts, deduplicates against what it already knows
  2. Surface — retrieves relevant memories at each prompt via keyword matching, with a token budget so it doesn't overwhelm the model
  3. Evaluate — tracks whether surfaced memories were followed, contradicted, or ignored
  4. Maintain — diagnoses each memory by effectiveness and proposes targeted fixes

Every memory gets classified into one of four quadrants based on how often it surfaces and how often it's followed:

High effectiveness Low effectiveness
Often surfaced Working — keep Leech — rewrite or escalate
Rarely surfaced Hidden gem — broaden keywords Noise — remove

The system also proposes its own improvements. When it notices that a keyword like "toml" causes irrelevant matches 93% of the time, it proposes de-prioritizing that keyword for future extractions. When it finds near-duplicate memories, it proposes consolidating them. All proposals go through user review before taking effect.

What it's like to use

In practice, I start a session and engram reminds me I can say /recall to load context from my last session. When I type a prompt, relevant memories surface inline. When the agent finishes, I get a quick triage summary if there are maintenance signals.

Periodically I run /memory-triage and spend a few minutes reviewing what engram has found: "these 6 memories are noise — remove all?", "this memory has 100% effectiveness but only surfaced twice — broaden keywords?". It takes a few minutes and the corpus gets noticeably better each time.

My favorite part is the adaptation pipeline. Engram watches its own feedback patterns and proposes system-level tuning — not just "fix this memory" but "stop using this keyword for extraction because it causes too many false positives." The system is learning how to learn better.

What about built-in memory?

Anthropic has introduced built-in auto-memory and dreaming features for Claude Code. Memory is definitely one of the primary problems that needs solving, and lots of smart people are working on it.

I tried auto-memory when it was introduced and found it unhelpful for the same reasons I'd had trouble with "please write this down somewhere" — non-deterministic recording, unreliable surfacing. I ended up turning it off because it was conflicting with engram. I haven't tried the new dreaming features yet — maybe they've materially improved things.

If the built-in memory management becomes good enough that most of engram is unnecessary, I'll be glad — less for me to maintain. Till then, I'm going to keep plugging away at this.

Where it's at

Engram is alpha. I've been using it daily for a few weeks and it's been genuinely useful, but APIs and data formats may still change. It's written in pure Go with no CGO dependencies, uses TF-IDF and BM25 for text matching, and calls the Anthropic API only for LLM-dependent operations like classification and extraction.

The whole thing is open source: github.com/toejough/engram

If you use Claude Code and have opinions about how memory should work for LLM agents, I'd love to hear from you. Issues and feedback are welcome.
